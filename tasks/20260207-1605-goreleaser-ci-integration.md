# PRD: GoReleaser CI Integration for Release Automation

**Created:** 2026-02-07 16:05 UTC

---

## Introduction

Implement automated release engineering using GoReleaser CI. The project will build and package Go binaries for macOS and Linux, attaching them as assets to official GitHub Releases triggered on tag pushes. This will streamline delivery, reduce manual effort, and ensure reproducible, cross-platform builds.

---

## Goals

- Enable one-click release creation via GitHub workflows
- Build binaries for macOS (amd64/arm64) and Linux (amd64/arm64)
- All binaries MUST be named `agent-smith`
- Attach built artifacts as downloadable release assets
- Auto-generate changelogs from git history per release
- Validate builds and ensure non-conflicting artifacts

---

## User Stories

- [ ] Story-001: As a maintainer, I want binaries for macOS and Linux to be attached to each GitHub Release so users can easily download and run them.

  **Acceptance Criteria:**
  - Releases are triggered by tag pushes
  - Assets for macOS (amd64, arm64) and Linux (amd64, arm64) are included
  - Checksums are attached for each artifact

  **Testing Criteria:**
  **Unit Tests:**
  - Validate .goreleaser.yml config logic for targets and artifacts
  - Verify release trigger matches tagged pushes

  **Integration Tests:**
  - End-to-end workflow: push tag, verify produced release with all assets

  **Component Browser Tests:**
  - Not applicable (backend only)

- [ ] Story-002: As a user, I want changelogs with each release so I can track what’s new.

  **Acceptance Criteria:**
  - Each release includes a changelog generated from commit history

  **Testing Criteria:**
  **Unit Tests:**
  - None—changelog is generated by GoReleaser tool

  **Integration Tests:**
  - Verify changelog appears correctly in GitHub Release body

  **Component Browser Tests:**
  - Not applicable

- [ ] Story-003: As a developer, I want to ensure releases are validated and artifacts are uniquely named per platform/version.

  **Acceptance Criteria:**
  - Naming conflicts are prevented
  - All artifacts have platform/version suffixes
  - Workflow runs `goreleaser check` as pre-validation

  **Testing Criteria:**
  **Unit Tests:**
  - Config validation for naming templates/suffixes

  **Integration Tests:**
  - CI produces unique assets for each platform/version

  **Component Browser Tests:**
  - Not applicable

---

## Functional Requirements

1. FR-1: The system MUST build binaries for macOS (amd64, arm64) and Linux (amd64, arm64) targets using GoReleaser, and MUST always name the binary `agent-smith` regardless of platform.
2. FR-2: Artifacts MUST be packaged (tar.gz or zip format) and checksums generated.
3. FR-3: GitHub Action MUST trigger workflow on tag pushes (any tag, semver recommended).
4. FR-4: Workflow MUST upload all artifacts and checksums as assets in GitHub Releases.
5. FR-5: Each Release MUST include an auto-generated changelog from git history.

---

## Non-Goals (Out of Scope)

- No Windows, Docker, Homebrew, or SBOM integrations in initial phase.
- No artifact signing or notarization for this setup.
- No custom release notes beyond auto-generated changelog.
- No multi-repo or monorepo support—single Go module only.
